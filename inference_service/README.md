# aaa_infrastucture

В данной папке реализован сервис для получения эмбеддингов от модели sergeyzh/rubert-mini-frida

## Этапы решения:
### Часть 1 - Выбор фреймворка
Я выбрал фреймворк - FastAPI по следующим причинам:
- Инференс трансформеров, даже таких компактных как rubert-mini-frida, занимает определенное время (десятки или сотни миллисекунд). Асинхронная архитектура FastAPI позволяет эффективно обрабатывать входящие HTTP-запросы, не блокируя event loop во время ожидания ответа от модели. Это критически важно для пропускной способности сервиса.
- При работе с текстовыми моделями важно контролировать входные данные. Pydantic позволяет легко описать структуру запроса (например, ограничить максимальную длину подаваемого текста, чтобы не превысить лимит токенов BERT), автоматически отсекая некорректные запросы до того, как они дойдут до ML-модели.
- FastAPI автоматически генерирует интерактивную документацию по стандарту OpenAPI. Это упрощает процесс тестирования API (можно отправлять тестовые тексты в модель прямо из браузера).

### Часть 2 - Реализация сервиса
- Реализован метод POST который принимает на вход строку, а возвращает эмбеддинг этой строки полученный от модели rubert-mini-frida.
- Модель загружается один раз при запуске
- Также в образе сразу загружена модель, что ускоряет развертку контейнера и не скачивает модель каждый раз, она запекается в образе и там хранится
- Реализован метод GET /health которые возвращает статус сервиса 
- Реализован запуск через Docker

### Часть 3 - Выбор метрик для бенчмарка

Были выбраны следующие метрики:
1) Latency P95 (Время ответа для 95% запросов)
   
Показывает общее время (отправка запроса клиентом -> обработка FastAPI -> инференс модели -> возврат ответа), за которое обслуживаются 95% самых быстрых запросов. В отличие от среднего времени (P50), метрика P95 позволяет увидеть реальную картину для подавляющего большинства.

Целевой порог: < 150-200 мс.

Обоснование: Модель mini-frida — это легковесная версия BERT. Она предназначена для быстрой обработки текста. Задержка до 200 мс воспринимается человеком практически как мгновенный ответ, что критично, если модель используется в чат-ботах или поиске.

Если значение будет выше, то можно попробовать:

    Поменять архитектуру
    
    Внедрить динамическую квантизацию модели (перевод весов из формата float32 в int8)
    
    Проверить сетевые задержки (возможно, проблема до того, как запрос попадает в контейнер)

2) Model Inference Time (Чистое время работы модели)
   
Время, затраченное исключительно на прогон данных через нейросеть (строки tokenizer(...) и model(**inputs)).

Целевой порог: < 50-80 мс.

Обоснование: Ожидаемое время инференса мини-моделей на современных CPU на один текст небольшой длины.

Если значение будет выше, то можно попробовать:

    Ограничить максимальную длину входного текста (max_length в токенезаторе), так как время работы трансформеров квадратично зависит от длины текста.
    
    Использовать батчинг - группировать несколько запросов в один, если запросы приходят одновременно.

3) Throughput (RPS - Requests Per Second)
 
Пропускная способность сервиса. Какое максимальное количество текстовых запросов сервис может успешно обработать за одну секунду до того, как начнет возвращать ошибки (например, 503 Service Unavailable) или бесконечно копить очередь.

Целевой порог: > 30-50 RPS (на 1 ядро CPU).

Обоснование: Учитывая время инференса в 50 мс, одно ядро математически может обработать около 20 запросов последовательно. Благодаря асинхронности FastAPI мы хотим выжать максимум из одного контейнера.

Если значение будет выше, то можно попробовать:

    Горизонтальное масштабирование: запустить несколько копий Docker-контейнера и поставить перед ними балансировщик нагрузки (например, Nginx).
    
    Запустить Uvicorn с несколькими воркерами (флаг --workers 4 вместо одного процесса по умолчанию).

4) Memory Usage (Потребление оперативной памяти RAM)

Объем оперативной памяти, который занимает запущенный Docker-контейнер. Для ML-моделей это критически важно, так как веса модели постоянно висят в памяти.

Целевой порог: < 450-550 МБ.

Обоснование: Сама модель rubert-mini-frida весит около 100-150 МБ. Добавляем накладные расходы Python, FastAPI и PyTorch. Удержание потребления до 500 МБ позволит разворачивать сервис на самых дешевых облачных серверах (с 1 ГБ RAM).

Если значение будет выше, то можно попробовать:

    Искать утечки памяти в коде — убедиться, что мы не сохраняем историю тензоров.
    
    Использовать более легковесные базовые образы Docker или перейти с PyTorch на ONNX Runtime, который потребляет значительно меньше оперативной памяти.

5) Fails (Error Rate, частота ошибок)
   
Процент запросов, на которые сервис ответил кодом ошибки (например, 500 Internal Server Error или 503 Service Unavailable).
Целевой порог: 0% (или < 0.1% для высоконагруженных систем).

Обоснование: Мы хотим чтобы у всех пользователей был положительный результат от взаимодействия с нашим сервисом.

Если значение будет выше, то можно попробовать:

    Смотреть логи FastAPI. Ошибки обычно значат, что либо мы передали кривой текст (не прошел валидацию Pydantic), либо не хватило оперативной памяти (OOM - Out of Memory) и контейнер «упал».

### Часть 4 — код, реализующий замер выбранных метрик
Реализован файл locustfile.py который позволяет имитировать нагрузку на сервис
Пример использования: 
pip install locust
locust -f locustfile.py --headless -u 20 -r 5 -t 30s --host http://127.0.0.1:8000
-u 20: пиковое количество одновременных пользователей (Users).
-r 5: скорость добавления пользователей в секунду (Rate).
-t 30s: продолжительность теста (30 секунд).
--host: адрес локального Docker-контейнера.

Результаты(проводился тест как в примере):
CPU USAGE - 152.78%
Memory Usage - 970.9MiB / 7.757GiB, поскольку в uvicorn стоит два воркера то было развернуто два независимых процесса -> весит 465МБ что удовлетворяет порогу
P95 - 610мс - выше порога, из-за эффекта очереди, если снизить количество юзеров одновременно до 5 (-u 5) то получается 160мс, что приемлемо, чтобы выдержать большую нагрузку нужно добавить воркеров, но в рамках работы в github codespace доступно только 2 ядра CPU и больше выжать не получается
Fails - 0(0.00%) - удовлетворяет порогу
RPS - 33.67 - удовлетворяет порогу
Model Inference Time AVG - 11 мc - удовлетворяет порогу
